## 3. UI/画面遷移 (Screen Flow)

### 画面構成

#### 🏠 画面A: ホーム画面

**必要な要素**

- 残高表示エリア（大きく目立つように）
- 「送金する」ボタン
- 「履歴を見る」ボタン

**アクション**

- 「送金する」→ 送金画面へ遷移
- 「履歴を見る」→ 履歴画面へ遷移

---

#### 💸 画面B: 送金画面

**必要な要素**

- 宛先選択エリア（ダミーユーザーのリストまたはドロップダウン）
- 金額入力フィールド
- 「確認する」ボタン
- 「キャンセル」ボタン

**アクション**

- 金額・宛先未入力時は「確認する」ボタンを無効化
- 「確認する」→ 送金確認画面へ遷移
- 「キャンセル」→ ホーム画面へ戻る

---

#### ✔️ 画面C: 送金確認画面

**必要な要素**

- 送金先ユーザー名表示
- 送金金額表示
- 送金後の残高予測表示
- 「送金する」ボタン
- 「戻る」ボタン

**アクション**

- 残高不足の場合はエラーメッセージを表示し「送金する」ボタンを無効化
- 「送金する」→ 送金処理を実行し、完了画面へ遷移
- 「戻る」→ 送金画面へ戻る

---

#### ✨ 画面D: 送金完了画面

**必要な要素**

- 完了メッセージ（「送金が完了しました」）
- 送金内容の要約（宛先・金額）
- 送金後の残高表示
- 「ホームに戻る」ボタン

**アクション**

- 「ホームに戻る」→ ホーム画面へ遷移

---

#### 📜 画面E: 履歴画面

**必要な要素**

- 履歴リスト（日時、宛先、金額を表示）
- 「ホームに戻る」ボタン
- 履歴が空の場合は「まだ送金履歴がありません」メッセージ

**アクション**

- 「ホームに戻る」→ ホーム画面へ遷移

---

### 画面遷移フロー

```
ホーム画面 (A)
  ├→ 送金画面 (B)
  │   └→ 送金確認画面 (C)
  │       └→ 送金完了画面 (D)
  │           └→ ホーム画面 (A)
  └→ 履歴画面 (E)
      └→ ホーム画面 (A)
```

## 4. 技術スタック (Tech Stack)

> 💡 新しい技術への挑戦は「1つ」に絞り、他は使い慣れたものを選ぶ

### Frontend

| 技術                           | 選定理由                                                           |
| ------------------------------ | ------------------------------------------------------------------ |
| **Next.js 14/15 (App Router)** | 現在のWeb開発の主流。Server Componentsなどの最新機能を学習するため |
| **TypeScript**                 | 金額計算のミスを防ぐ「型安全性」が金融系開発で必須なため           |
| **Tailwind CSS**               | クラス名を付ける時間を節約し、爆速でスマホUIを作るため             |
| **Lucide React**               | モダンで統一感のあるアイコンを手軽に使うため                       |

### Backend

| 技術                       | 選定理由                                             |
| -------------------------- | ---------------------------------------------------- |
| **Next.js Route Handlers** | Next.js内にAPIを作成。別途サーバーを立てる手間を削減 |

### Database

| 技術             | 選定理由                                                          |
| ---------------- | ----------------------------------------------------------------- |
| **SQLite**       | サーバー構築不要。ファイル1つで動くため環境構築が最速             |
| **Prisma (ORM)** | SQLを書かずにDB操作が可能。後でSupabase(PostgreSQL)への移行も容易 |

### Dev Tools

| 技術               | 選定理由                                                                  |
| ------------------ | ------------------------------------------------------------------------- |
| **GitHub Copilot** | (Student Pack特典) 未知の技術(Next.js)の学習・実装スピードを3倍にするため |

### 🎯 今回の技術チャレンジ

**Next.js 14/15 (App Router) + Server Components**

- 従来のReact開発とは異なるパラダイムを学習し、最新のWeb開発手法を習得する

### 2. PRD (製品要件ドキュメント) と チーム開発手法

PRDに関してはGoogleが「これが公式テンプレートです」として公開している単一のページはありません（社内文化として浸透しているものです）。ただし、Googleが社内の「働き方」や「チームビルディング」の手法を公開しているサイトがあり、そこで根本的な考え方を学ぶことができます。

- **サイト名**: Google re:Work（リワーク）
- **リンク**: [https://rework.withgoogle.com/intl/jp/](https://rework.withgoogle.com/intl/jp/)
  - Googleが実践している「効果的なチームの作り方」や「目標設定（OKR）」などのガイドが日本語で読めます

#### 補足：PRDの書き方を詳しく知りたい場合

Google出身のプロダクトマネージャーなどが推奨する一般的なPRDの形式については、シリコンバレーのプロダクトマネジメント情報の総本山である**SVPG (Silicon Valley Product Group)**の記事や、Atlassian社の解説が業界標準としてよく参照されます。
